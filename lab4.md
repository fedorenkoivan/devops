Загальна архітектура системи
Розгортаємо той же застосунок, що і в Лабораторній роботі №1. В цій роботі замість одного вузла система тепер розгортається на 2 окремі віртуальні машини в межах однієї мережі.

Архітектура системи:
          +--------------VM1 (worker)---------------+    +---VM2 (db)---+
client → | nginx (reverse proxy) → web application | → | SQL database |
          +-----------------------------------------+    +--------------+

Розгортання сервісу
Для виконання роботи необхідно розгорнути веб-застосунок, базу даних та reverse-proxy на віртуальних машинах з Linux.
Розгортання повинно відбуватися в 2 етапи:
Provisioning (Terraform)
Необхідно описати інфраструктуру в .tf файлах:
    Використовуйте провайдер libvirt (для KVM/QEMU) або virtualbox.
    Створення двох ВМ на базі офіційних Cloud-образів (Ubuntu/Debian/CentOS).
    Використання cloud-init для базового налаштування (додавання SSH-ключа студента).

Історія комітів повинна бути логічною та зрозумілою, відображати хід виконання роботи студентом.

Репозиторій повинен містити документацію по виконаному завданню. Точкою входу в документацію має бути файл README.md або README.rst в корені репозиторію. За потреби можна створювати додаткові файли документації у директорії docs, та посилатися на них із файлу README. Документація повинна містити інструкції по запуску terraform та ansible
Критерії приймання роботи
Автоматизація: Інфраструктура (2 ВМ) піднімається однією командою terraform apply. Налаштування інфраструктури виконується одним запуском ansible-playbook.
Ідемпотентність: Повторний запуск ansible-playbook не повинен призводити до змін, якщо конфігурація вже відповідає опису (окрім специфічних випадків, як-от перевірка статусу).
Декларативність: Відсутність прямих викликів shell або command в Ansible, де існують стандартні модулі (apt, user, template, systemd).
Розподіленість: Застосунок на VM(worker) успішно підключається до БД на VM(db). Прямий доступ до БД ззовні заблокований.
Користувачі: Доступ за SSH працює для ansible, teacher, operator. Користувач operator має обмежені права sudo тільки на перезапуск сервісів.
Health Checks: Ендпоінти /health/alive та /health/ready повертають коректні статуси (ready має перевіряти зв'язок між VM1(worker) та VM2(db)).
